# Java RMI (Remote Method Invocation)

O **Java RMI (Remote Method Invocation)** é um mecanismo da plataforma Java que permite a implementação de aplicações distribuídas de forma transparente, seguindo o paradigma de **invocação remota de métodos (RMI)**, análogo ao RPC (Remote Procedure Call), mas em um contexto orientado a objetos. Seu objetivo principal é permitir que um programa Java em execução em uma JVM (Java Virtual Machine) invoque métodos de um objeto localizado em outra JVM, como se fosse uma chamada local. Essa abstração encapsula a complexidade da comunicação via sockets, serialização de parâmetros e tratamento de erros de rede.

A arquitetura do RMI é composta por várias camadas que trabalham em conjunto para fornecer a transparência de localidade. A camada de **stub e skeleton** atua como proxy para o objeto remoto. O **stub** reside no cliente e é responsável por marshaling (serialização) dos argumentos da chamada, comunicação com o servidor e unmarshaling do resultado. O **skeleton** reside no servidor, recebe a requisição, faz o unmarshaling, invoca o método real no objeto de implementação e envia o resultado de volta para o stub. Abaixo deles, a camada de referência remota (Remote Reference Layer) lida com a semântica da invocação (p.ex., unicast, multicast) e a camada de transporte (Transport Layer) estabelece a conexão física entre as JVMs, lidando com TCP/IP.

O desenvolvimento de um sistema usando RMI segue uma sequência definida de passos. Primeiro, deve-se definir uma **interface remota** que estende a interface `java.rmi.Remote`. Todos os métodos nesta interface devem declarar lançar `java.rmi.RemoteException`. Esta interface define o contrato que o objeto remoto oferece. Em seguida, é necessária uma **implementação concreta** dessa interface. Uma prática comum é estender a classe `UnicastRemoteObject`, que automaticamente "exporta" o objeto, tornando-o pronto para receber chamadas remotas em uma porta TCP.

Para que os clientes possam localizar o serviço remoto, um serviço de nomes é utilizado: o **RMI Registry (`rmiregistry`)**. Ele é um daemon que executa em uma porta conhecida (por padrão, a 1099) e mantém um mapeamento entre nomes (strings) e referências para objetos remotos. O servidor registra sua implementação no registry usando o método `rebind()` (ou `bind()`) da classe `Naming` ou diretamente através da API `Registry`. O cliente, por sua vez, utiliza o método `lookup()` para obter a referência (o stub) do objeto remoto, a partir do nome registrado.

O processo de compilação e execução envolve ferramentas específicas. O compilador `rmic` (agora considerado defasado, pois a geração dinâmica de stubs via reflexão é preferida) era usado para gerar as classes stub e skeleton a partir da classe de implementação. Para executar o sistema, inicia-se o `rmiregistry`, depois o servidor que registra o objeto e, finalmente, o cliente que realiza a busca e as invocações.

Uma característica poderosa do RMI é a **passagem de objetos como parâmetros**. Diferente de RPCs tradicionais, que tipicamente suportam apenas dados primitivos, o RMI permite que objetos completos sejam passados para métodos remotos. Para isso, o objeto deve implementar a interface `java.io.Serializable`. Quando um objeto serializável é passado como argumento, ele é **serializado (marshaling)** no lado do cliente e **desserializado (unmarshaling)** no lado do servidor, efetuando uma **passagem por valor**. Isso possibilita um design mais flexível, como o padrão "**Mobile Code**" ou "**Code on Demand**", onde a lógica (um objeto `Task` no exemplo dos slides) é enviada para ser executada no servidor, que devolve apenas o resultado.

O RMI também enfrenta desafios em ambientes de rede restritivos. **Firewalls** frequentemente bloqueiam portas TCP não padrão. Como o registry usa a porta 1099, mas os objetos remotos podem ser exportados em portas efêmeras (aleatórias), a comunicação pode ser bloqueada. As soluções para isso incluem:

- Configurar um **port range fixo** para a exportação de objetos usando `UnicastRemoteObject.exportObject(Remote obj, int port)`.
    
- Utilizar uma **RMISocketFactory personalizada** que encapsule o tráfego RMI dentro de túneis **HTTP ou HTTPS**, contornando as restrições do firewall ao usar as portas 80 ou 443. Isso resulta em chamadas RMI encapsuladas em pacotes HTTP (HTTP-tunneling).
    

Em termos de comparação com outros paradigmas, o RMI se destaca por sua **integração natural com a linguagem Java**, oferecendo transparência e simplicidade no desenvolvimento. Suas **vantagens** incluem a portabilidade entre qualquer plataforma com uma JVM, o suporte à mobilidade de código e a abstração completa da comunicação de rede. Como **desvantagens**, pode-se citar seu caráter **proprietário** (é uma tecnologia exclusiva do ecossistema Java) e os desafios de configuração em redes com firewalls, embora existam soluções para contorná-los.

Portanto, o Java RMI é um framework robusto e elegante para a construção de sistemas distribuídos homogêneos (em Java), implementando de forma eficaz o modelo de objetos distribuídos e escondendo a complexidade inerente à comunicação entre processos em rede.
